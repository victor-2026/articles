# Переход от вайб- к вайп-кодингу: benefits, challenges and risks

**Last Updated:** 2026-05-10
**Sources:** wiki/vibe-wipe-coding-guide.md, raw/vibe-coding-links.md

---

## Введение

### Определения

**Vibe Coding** — создание программного обеспечения через естественный язык с AI ассистентом. Программист описывает "вибрацию" того что хочет, AI генерирует код. Работает отлично для прототипирования, но недостаточно для production систем.

**Wipe-coding** (или Flow-state coding) — это стиль разработки, при котором программист перестаёт писать код «руками» построчно. AI «стирает» старые абстракции и «накатывает» (wipe) новые слои кода сразу во многих файлах. Появление автономных AI-агентов (Windsurf, Cursor, Claude Code, Bolt) знаменует переход от «чата с подсказками» к полноценному AI-напарнику с «руками» в системе.

---

## 1. Benefits (Преимущества)

### 1.1 Скорость и эффективность
- **Ускорение в 3 раза** — Buzzhive Sandbox пример показывает реальное ускорение процесса разработки
- **Mass refactoring** — Изменение сразу во многих файлах без ручной работы
- **Параллельные агенты** — Разведка API и генерация документации одновременно

### 1.2 Flow-state разработка
- Программист фокусируется на архитектуре, не на синтаксисе
- AI берёт на себя рутинные задачи
- Можно оставаться в состоянии потока (flow) дольше

### 1.3 Автономность AI-агентов
- AI видит весь контекст проекта
- AI индексирует все файлы
- AI имеет доступ к терминалу
- AI самостоятельно запускает npm install, pytest
- AI самостоятельно исправляет ошибки

### 1.4 Pipeline Triad Pattern
```
Creator → Critic → Arbiter
   ↑_________|_________|
```
Agentic подход для production AI разработки с Human Gates на ключевых этапах.

---

## 2. Challenges (Вызовы)

### 2.1 Потеря контекста
- AI видит весь проект, но может потерять нюансы
- Сложность понимания бизнес-логики вне явного кода
- Риск "hallucination" при работе с большим контекстом

### 2.2 Review burden
- Человек должен проверять каждое изменение
- Необходимость проверять что AI не "убежал" в неправильном направлении
- Требуется экспертиза для оценки качества изменений

### 2.3 Test-driven loop
- AI итерирует пока тесты не пройдут
- Может потребоваться много итераций
- Нужны надёжные тесты для обратной связи

### 2.4 Границы изменений
- Что можно/нельзя менять — нужно чётко задать
- Требуется явное описание ограничений
- Риск случайного изменения критических компонентов

---

## 3. Risks (Риски)

### 3.1 Технические риски
- **AI runs wild** — Бесконтрольные изменения без review
- **Security gaps** — no rate limiting, IDOR уязвимости, exposed API keys
- **No CI/CD** — Теряется safety net, сложно откатить изменения

### 3.2 Knowledge risks
- **Comprehension debt** — Код непонятен без AI-контекста
- Сложность поддержки без AI-ассистента
- Зависимость от AI инструментов

### 3.3 Operational risks
- **No monitoring** — Проблемы узнаём от пользователей, не из алертов
- Технический долг накапливается незаметно
- Сложность аудита и compliance

---

## 4. Экономика Wipe-coding

| Метрика | Значение |
|---------|-----------|
| Human time per task | ~1 час |
| API cost per task | $6-12 |
| Acceleration | ~3x |

---

## 5. Когда использовать

| Vibe Coding | Wipe-coding |
|-------------|-------------|
| Early exploration | Production systems |
| MVP/prototypes | Team collaboration |
| Testing ideas | Compliance requirements |
| Getting unstuck | Long-term maintenance |

---

## 6. Best Practices для Wipe-coding

1. **Set clear boundaries** — Что можно/нельзя менять
2. **Review every change** — Не давать AI бесконтрольно работать
3. **Test-driven** — AI фиксит пока тесты не пройдут
4. **Keep artifacts** — Документировать решения для будущего
5. **CI/CD from day one** — Иначе столкнёшься с техническим долгом

---

## 7. Вывод: Не either/or — Yes And

Оба подхода имеют ценность:
- **Vibe coding** — раскрывает импульс, искру, позволяет тестировать идеи
- **Wipe-coding** — замедляет для ясного мышления, создаёт артефакты

Используй Vibe coding для прототипирования. Используй Wipe-coding для production.
