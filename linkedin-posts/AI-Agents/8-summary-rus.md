# Статья 8: Скиллы — это не npm-пакеты 🏗️

**Продолжение эксперимента из Части 1: 0% lift → почему это произошло — и что я построил вместо этого.**

**Проблема:** Когда один и тот же LLM запускает скилл и оценивает его результат — это как студент, который сам себе ставит оценку. Без внешнего судьи скиллы производят «уверенный мусор» (confident garbage).

**Что подсмотрел у других:**
- **Антон Гулин** — правильный тезис (eval-driven, shareable), но его реализация tools-first: MCP-серверы, Copilot. Он усиливает человека-тестировщика.
- **Hack'n'Vibe** — RULES.md: 9 стоп-правил, которые говорят не «как делать хорошо», а «что ломается каждый раз». Агент учится на реальных падениях через learn-механизм.
- **Rinat Abdullin** — BDD (Given-When-Then) как исполняемая валидация. Spec-Driven Development = мёртвые спеки, а BDD = спецификация, которая ломает билд при расхождении.


**Архитектура (3 слоя вместо одного):**

```
Context Layer     — AGENTS.md (границы) + Stop-rules (антипаттерны)
Skill Layer       — SKILL.md + Self-Evaluation (4 оси, без внешнего судьи)
Validation Layer  — BDD Harness (нарушение → ошибка сборки, а не устаревший док)
```

**4 оси Self-Evaluation:** Correctness (компилируется?), Relevance (по задаче?), Stability (стабильность вывода?), Coverage (крайние случаи?).

**Сравнение с Гулиным:**
- Гулин: AI = инструмент для человека, скилл = код-ген, валидация = внешний судья
- Мы: AI = агент с guardrails, скилл = поведенческий blueprint, валидация = self-eval + BDD

**Цифры:** 8 скиллов, 4 оси, 9 стоп-правил. Всё на одной модели без внешнего судьи. Скиллы — это не генераторы кода, а поведенческие чертежи с контекстом, самопроверкой и исполняемой валидацией.
